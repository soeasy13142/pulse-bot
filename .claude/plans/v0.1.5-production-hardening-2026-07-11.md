# v0.1.5 Production Hardening — Design Spec

> **Status**: Draft (brainstorming approved)
> **Date**: 2026-07-11
> **Target release**: v0.1.5 (patch release on top of v0.1)
> **Author**: brainstormed via superpowers:brainstorming, scope: B1 + B2 + B3 + B4 from v0.2 candidate catalog

## Context

Pulse Bot v0.1 ships with 73 tests, 90% coverage, and a working Telegram → vault pipeline. Production deployment on a VPS has surfaced four reliability gaps that block "leave it running for months" confidence:

| # | Gap | Symptom |
|---|-----|---------|
| **B1** | No graceful shutdown | systemd restart loses in-flight Telegram messages (SIGKILL mid-handler) |
| **B2** | Unstructured logs | Hard to grep, correlate across requests, or feed to log aggregation |
| **B3** | No watchdog | bot can hang indefinitely without systemd noticing (default `Restart=on-failure` only catches exit code) |
| **B4** | Silent DLQ accumulation | Failed pushes pile up in `dead_letter.jsonl`; nobody is told until manual `tail` |

All four are **S-grade** (1-2 days each), **independently valuable**, but **loosely coupled** (B1↔B3 tight, B2↔B4 loose). This spec bundles them as a single atomic v0.1.5 release to avoid breaking changes across multiple upgrades.

## Goals

- Eliminate message loss on systemd restart.
- Make bot logs machine-parseable (JSON) without breaking human readability.
- Have systemd detect and restart a hung bot within 30s.
- Alert the bot owner within minutes of any DLQ entry.
- Keep test coverage ≥ 90%.
- Stay within the project's KISS / DRY / YAGNI discipline (no premature abstractions).

## Non-Goals

- Full Prometheus metrics endpoint (deferred to v0.2 — overkill for single-user bot).
- Distributed tracing / OpenTelemetry (no multi-service story).
- LLM-assisted intent classification (separate v0.2 work).
- Multiple notification channels (email, Slack) — Telegram-only is enough for v0.1.5.

## Feature B1: Graceful shutdown

### Problem
`application.run_polling()` blocks; on `systemctl restart`, the process receives SIGTERM and python-telegram-bot's runtime does drain, but our `on_message` and `on_command` handlers don't track their own completion. Result: messages being processed at restart time can be lost (acknowledged to Telegram, but card write / git push incomplete).

### Solution
Introduce `pulse_bot.lifecycle.ShutdownCoordinator` — a per-process counter + asyncio.Event that tracks in-flight handler coroutines.

```python
class ShutdownCoordinator:
    def __init__(self, drain_timeout: float = 30.0) -> None:
        self._inflight = 0
        self._event = asyncio.Event()
        self._drain_timeout = drain_timeout
        self._shutting_down = False

    @property
    def is_shutting_down(self) -> bool:
        return self._shutting_down

    async def track(self, coro) -> Any:
        if self._shutting_down:
            raise ShutdownInProgress("rejecting new work")
        self._inflight += 1
        try:
            return await coro
        finally:
            self._inflight -= 1
            if self._inflight == 0:
                self._event.set()

    async def wait_drain(self) -> bool:
        if self._inflight == 0:
            return True
        try:
            await asyncio.wait_for(self._event.wait(), timeout=self._drain_timeout)
            return True
        except asyncio.TimeoutError:
            return False

    def request_shutdown(self) -> None:
        self._shutting_down = True
```

### Integration
- `bot.py` wraps each handler entry with `await coordinator.track(handler(update, context))`.
- New handlers receive the coordinator via `context.bot_data["coordinator"]`.
- Signal handling uses asyncio-native API: `loop.add_signal_handler(signal.SIGTERM, coordinator.request_shutdown)`. Avoid `signal.signal()` (sync) since `application.run_polling()` runs in an asyncio loop and sync signal handlers can race with coroutine resumes.
- Main loop after `run_polling()` returns: call `coordinator.wait_drain()`; on success `sys.exit(0)`, on timeout `os._exit(1)` (so systemd records dirty shutdown).

### Edge cases
- New messages arriving during shutdown: rejected with `ShutdownInProgress`; user sees `⚠ Bot restarting, try again in a moment`.
- Cooldown race: shutdown requested while drain in progress → idempotent.
- Signal received before `run_polling()` starts: handler registered, flag set, polling starts → immediately exits.

## Feature B2: Structured logging

### Problem
Current code uses `logging.basicConfig()` with text format. journalctl collects the output but correlation across handlers / messages requires manual parsing.

### Solution
JSON output via `python-json-logger`, configurable via `LOG_FORMAT` env (default `json`, fallback `text` for local dev without journald).

```python
# pulse_bot/observability.py
import logging
import sys

def setup_logging(level: str = "INFO", fmt: str = "json") -> None:
    handler = logging.StreamHandler(sys.stderr)
    if fmt == "json":
        from pythonjsonlogger import jsonlogger
        formatter = jsonlogger.JsonFormatter(
            "%(asctime)s %(levelname)s %(name)s %(message)s",
            rename_fields={"asctime": "ts", "levelname": "level", "name": "logger"},
        )
    else:
        formatter = logging.Formatter("%(asctime)s [%(levelname)s] %(name)s: %(message)s")
    handler.setFormatter(formatter)
    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(getattr(logging, level.upper(), logging.INFO))
```

### Context propagation
Handlers add `request_id` (per Telegram update) and `user_id` via `logger.info("...", extra={...})` pattern. Optional: use `contextvars` to auto-attach (deferred — adds complexity for marginal value).

### Field schema

| Field | Source | Example |
|---|---|---|
| `ts` | formatter | `2026-07-11T16:30:00Z` |
| `level` | formatter | `INFO` |
| `logger` | formatter | `pulse_bot.bot` |
| `message` | log call | `Captured card 2026-07-11_223045_想法` |
| `request_id` (opt) | `extra=` | `tg-update-12345` |
| `user_id` (opt) | `extra=` | `123456789` |
| `card_path` (opt) | `extra=` | `/opt/pulse-bot/vault/00_Inbox/_pulse/...md` |

## Feature B3: systemd watchdog + healthcheck

### Problem
Default `Restart=on-failure` only fires on exit code ≠ 0. A bot deadlocked in a `while True:` loop runs forever with no signal to systemd.

### Solution
Periodic `sd_notify("WATCHDOG=1")` ping; systemd kills the process if pings stop arriving within `WatchdogSec=`.

```python
# pulse_bot/observability.py
import os
import threading
import time

class WatchdogPinger:
    def __init__(self, interval: float = 10.0) -> None:
        self._interval = interval
        self._stop = threading.Event()
        self._enabled = "NOTIFY_SOCKET" in os.environ
        self._thread: threading.Thread | None = None

    def start(self) -> None:
        if not self._enabled:
            logger.info("watchdog disabled (NOTIFY_SOCKET not set)")
            return
        self._thread = threading.Thread(target=self._loop, daemon=True)
        self._thread.start()

    def stop(self) -> None:
        self._stop.set()
        if self._thread:
            self._thread.join(timeout=2.0)

    def _loop(self) -> None:
        import sdnotify
        n = sdnotify.SystemdNotifier()
        while not self._stop.wait(self._interval):
            n.notify("WATCHDOG=1")
```

### Systemd unit changes

```ini
[Service]
Type=notify                # changed from default (simple)
WatchdogSec=30             # new
```

Ping interval = 10s, watchdog timeout = 30s → tolerates 2 missed pings before kill. Conservative; can tighten to 5s/15s in v0.2 if observability shows false-positives.

### Healthcheck endpoint (deferred)
A localhost:8080 `/health` HTTP endpoint is **not in v0.1.5**. Rationale: single-user bot, systemd watchdog covers the hang case, HTTP adds surface area (port management, firewall). If external monitoring is wanted later, can be added without breaking changes.

## Feature B4: Dead-letter auto-alert

### Problem
`dead_letter.jsonl` accumulates failed push attempts. No mechanism tells the owner; failures are silent until manual inspection.

### Solution
Hook into `DeadLetterQueue.append()` with an event callback. On entry: check threshold, fire Telegram notification if exceeded.

```python
# pulse_bot/dead_letter.py (modified)
class DeadLetterQueue:
    def __init__(self, path: Path, on_new_entry: Callable[[], None] | None = None) -> None:
        self.path = path
        self._on_new_entry = on_new_entry

    def append(self, card_path: Path, error: str, payload: dict) -> None:
        # ... existing write logic ...
        if self._on_new_entry:
            try:
                self._on_new_entry()
            except Exception:
                logger.exception("dead_letter on_new_entry callback failed")
```

### Alert logic (in `bot.py`)
On bot startup AND every time `on_new_entry` fires:

1. Count current DLQ entries.
2. If count ≥ `DLQ_ALERT_THRESHOLD` AND (no alert sent in last `DLQ_ALERT_COOLDOWN` seconds) → send Telegram message to first whitelisted user.
3. Message format:
   ```
   ⚠ Dead-letter queue alert
   File: /opt/pulse-bot/dead_letter.jsonl
   Entries: 7
   Last 3 errors:
     - 2026-07-11T16:00:00 push failed: Host key verification failed
     - ...
   Inspect: sudo tail -20 /opt/pulse-bot/dead_letter.jsonl
   Replay: see docs/runbook.md#f7-死信队列
   ```

### Edge cases
- Bot restart while DLQ non-empty: alerts on startup if over threshold.
- Threshold cooldown: tracked in-memory only (acceptable — bot restart resets, that's fine).
- `on_new_entry` callback fails: caught and logged; DLQ write succeeds regardless (don't let alerting break persistence).
- Owner not whitelisted (config error): alert attempt logs error and gives up silently.

## Testing strategy

| File | Covers | Type |
|---|---|---|
| `tests/test_lifecycle.py` | B1: ShutdownCoordinator unit + signal handler integration | unit |
| `tests/test_observability.py` | B2 (JSON formatter) + B3 (WatchdogPinger) | unit |
| `tests/test_dead_letter_alert.py` | B4: DLQ hook + alert trigger + cooldown | unit |
| `tests/test_integration.py` (extend) | full SIGTERM → drain → exit; systemd unit dry-run | integration |

Coverage target: ≥ 90% (current 90%, do not regress).

### Test patterns
- `monkeypatch.setattr(bot_mod, "sdnotify", FakeNotify)` for watchdog.
- `capfd` or `capsys` to capture stderr; assert JSON keys present.
- `monkeypatch.setattr(time, "time", lambda: <faked>)` for cooldown timing — avoid new `freezegun` dependency.
- `tmp_path` for DLQ fixture (already established pattern).

## Migration / rollout

### For users upgrading v0.1 → v0.1.5

```bash
# 1. Pull + install deps
cd /opt/pulse-bot/app
sudo -u pulse-bot git pull
# requirements.txt now includes python-json-logger and sdnotify; the
# pip install below picks them up:
sudo -u pulse-bot .venv/bin/pip install -r requirements.txt

# 2. Update systemd unit
sudo systemctl edit pulse-bot
# Add:
# [Service]
# Type=notify
# WatchdogSec=30
sudo systemctl daemon-reload

# 3. Restart
sudo systemctl restart pulse-bot

# 4. Verify
sudo journalctl -u pulse-bot -f | grep -i watchdog
# Expect: every ~10s a line "watchdog ping sent" (or similar)
```

### Rollback
```bash
git checkout v0.1
sudo -u pulse-bot .venv/bin/pip install -r requirements.txt
sudo systemctl revert pulse-bot   # reverts the edit
sudo systemctl restart pulse-bot
```

## Acceptance criteria

- [ ] `pytest --cov=pulse_bot tests/` ≥ 90% coverage, 73+ tests + new ones passing.
- [ ] SIGTERM during in-flight message: that message completes successfully before exit.
- [ ] All `logging.info(...)` calls produce valid JSON when `LOG_FORMAT=json`.
- [ ] `journalctl -u pulse-bot -f` shows watchdog ping every ~10s.
- [ ] Killing bot process without graceful shutdown → systemd restarts within 30s of hang.
- [ ] Filling DLQ with 5 entries triggers one Telegram alert; 6th entry within 1 hour does not re-alert.
- [ ] No regressions in v0.1 features (capture, push, retry, /recent, /start, /p).
- [ ] `docs/deployment.md` and `docs/runbook.md` updated with v0.1.5 upgrade steps.

## Open risks

| Risk | Mitigation |
|---|---|
| `python-json-logger` breaks across major Python | Pin version in `requirements.txt`; test on 3.11 (project standard) |
| `sdnotify` ctypes call fails silently on non-systemd system | `NOTIFY_SOCKET` env guard; WatchdogPinger.start() logs and returns |
| Cooldown resets on bot restart → alerts on every restart if DLQ is dirty | Acceptable; startup alert is desirable anyway |
| Signal handler races with `application.run_polling()` | Already addressed in B1 integration — uses `loop.add_signal_handler()`, not `signal.signal()` |

## Out of scope (deferred to v0.2+)

- HTTP `/health` endpoint
- Prometheus `/metrics` endpoint
- Log shipping (Loki, Elasticsearch)
- Multi-channel alerting (email, Slack)
- LLM-assisted intent classification
- Multi-vault support
- `/promote`, `/find`, `/remind`, image attachment — all v0.2 candidate features
